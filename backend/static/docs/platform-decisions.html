Decision memo · 2026-07-05

Where we land the database, where the admin app goes, and what we ship first.

Two decisions locked, one rollout sequenced. So neither becomes tribal memory.

Scope DB strategy + admin IA Horizon 12 weeks Team size 1–3 engineers Current scale ~1k users · <200 GB · low-hundreds QPS
Decisions locked

Two things we're committing to now.

Both fell out cleanly from the research; both have well-defined "trip" conditions if the assumptions shift. The rest of this memo is the reasoning and the punch list.

Database

Stay on RDS PostgreSQL. Move off Single-AZ t4g burstable to Multi-AZ m6g.large. Buy the Reserved Instance after 90 days at size.

"Sustainable dependable" for a founder-led team means the DB must never be the thing that wakes someone up, and recovery must be a button, not a runbook. RDS Multi-AZ delivers exactly that under a 99.95% SLA — a ~35× TCO penalty argument against every self-managed alternative once labor is priced honestly.

~$263/mo on-demand → ~$171/mo on 1-year RI
Admin IA

Progressive Convergence. Ship net-new value in the admin app in ~2 weeks; migrate console SYSTEM pages in waves behind a per-page feature flag.

The only proposal that ships user-visible value fast and commits to the correct end state — zero SYSTEM-only pages in console — without the ten-engineer-week upfront bill of a big-bang migration. Every migration is individually revertible.

P1 in ~2 weeks · full convergence by week 12
Precedent set this session. The Agents-tab migration (admin PR #7, console PR #164) is the first wave of Progressive Convergence in flight — it's how every subsequent SYSTEM page should move. Console keeps the file on disk for one release cycle so a stale bookmark hits a friendly not-found; nav entry + route are pulled the moment the admin tab is live.
02 · Database strategy

Eight options were priced. One is the answer.

The honest answer for a production SaaS at this scale is: stay on RDS Postgres, but stop pretending Single-AZ t4g burstable is a production posture. The compute-vs-labor math against self-managed alternatives is decisive — infra numbers look cheaper, all-in numbers are 15–20× worse once senior-SRE time is priced. The modern-managed category (Neon, Supabase, Crunchy, Timescale) is real but structurally more expensive at this scale unless we're paying for something they uniquely bundle.

Option Monthly Ops burden Verdict Reason
RDS PostgreSQL db.m6g.large Multi-AZ · gp3 · 30d PITR $263or $171 on 1yr RI ~2 hrs/mo · <$300 Pick Managed failover 60–120 s, PITR is one API call, Blue/Green for majors. AWS is on the hook at 3am.
Aurora PostgreSQL r6g.large writer + reader · 100 GB $443baseline ~2 hrs/mo · <$300 Defer Sub-30 s RTO + readable replica cost $180/mo more for benefits not yet needed. Right answer at 500 GB or on a tighter SLA.
Self-managed EC2 Patroni 3-node · WAL-G to S3 $332infra · $4.8k labor ~40 hrs/mo · $4,800 Skip Saves ~$70–130/mo on infra, costs $4,500/mo more all-in. Break-even needs a dedicated Postgres SRE.
In-cluster CNPG 3× on shared EKS · Barman to S3 $95–310infra · $4.3k labor ~36 hrs/mo · $4,320 Skip Forced Barman-plugin migration in 2026; zonal-EBS gotcha turns node evictions into 15–30 min outages. Only wins amortizing across many instances.
Crunchy Bridge Standard-8 HA · same region $290→ $890 at 500 GB + replica ~2 hrs/mo Defer Cleanest match to the "boring Postgres" brief after RDS — free egress, PgBouncer bundled, superuser. Lose only for lack of one-throat-to-choke with AWS.
Neon Scale 2 CU always-on · PrivateLink $360→ $835 at 500 GB ~2 hrs/mo Defer Real branching for CI/preview envs is a genuine productivity win. May 2025 control-plane fragility knocks it behind Crunchy for prod.
Supabase Team Team + Small · PITR add-on $725→ $1,067 with replica ~2 hrs/mo Skip $599 Team floor makes it structurally uncompetitive as pure DB. Only worth it if we're retiring in-house auth or realtime onto their stack.
Timescale Cloud Performance HA $330–370→ $1,050+ at scale ~2 hrs/mo Skip Paying the Timescale premium for an extension we don't call. Wrong tool at this scale.

When we reconsider

These are the conditions that flip the decision. None are true yet. All are worth setting an alert on.

  • Aurora when data crosses ~500 GB, or SLA tightens to <60 s RTO, or Blue/Green upgrades become routine.
  • Crunchy when AWS egress-to-app becomes a line item, we want superuser + a smaller vendor's support bench, and we don't need branching.
  • Neon when per-PR preview DBs would meaningfully accelerate the team AND we're willing to accept 2025-era control-plane risk.
  • Self-managed when a dedicated Postgres SRE joins (sunk-cost time), or RDS blocks a load-bearing extension (TimescaleDB OSS, custom C), or data clears ~2–3 TB.
Untested claim: the current live RDS instance class and IOPS profile were not verified from the console. Confirm real Performance Insights numbers before committing to m6g.large — the recommendation assumes low-hundreds QPS, <200 GB, no sustained IOPS saturation. Also: quarterly PITR restore-to-scratch drill goes on the calendar. AWS does not test your backups for you, and an untested backup is not a backup.

04 · Rollout & P1 punch list

What happens in what order.

Four phases; each is bounded, individually revertible, and gates the next. Phase 0 is halfway done — the aws-dashboard rename to admin.clue2.app happened this session, along with the Google-OAuth edge + FastAPI BFF holding the SYSTEM JWT server-side. What remains is DNS at the apex and finishing the Daari secrets-binding path.

Phase 0 · Rehome Week 1 · in flight
  • Repo rename aws-dashboard → platform-admin-dashboard
  • Deploy at admin.clue2.app in control namespace, reusing core-sa and aws-iam-credentials
  • Legacy aws-dashboard.clue2.app 301 redirect for 90 days
  • Google-OAuth at edge, @clue2solve.com allowlist
  • FastAPI BFF holds long-lived SYSTEM JWT server-side — frontend never sees it
  • Emergency-admin kubectl-secret break-glass path in BFF
  • Cookie Domain=.clue2.app for cross-surface SSO with console
Phase 1 · Ship net-new Weeks 1–2
  • Ship P1 sections: Overview · Cost · Accounts (RO) · Administration · Credentials health · Diagnostics
  • Embed aws-dashboard Services/Access/Cluster/Compute tabs as route-mounts under Platform Infrastructure — nothing regresses
  • Zero console changes except one link at top of SYSTEM nav
  • Explicit security review at end of P1: every BFF route asserts user_type=SYSTEM, cross-account queries can't leak, impersonation design signed off before P2 build
Phase 2 · First migration wave Weeks 3–6 · per-page cutover
  • Pages: Accounts CRUD · Agents · Audit · KubernetesClusters · ClusterStacks · ClusterStores · Builders · PlatformLimits. aws-dashboard tabs absorbed as first-class
  • Per-page cutover cycle (~4 weeks each, staggered): parity build → both-live (staff dogfood, external still on console) → flip (console soft-redirect banner with query string preserved) → delete (git rm + nginx 301)
  • Reversal at any stage: flip flag false → console page reappears. Deleted-code recovery within 2-week cooldown via git revert
  • Extract shared @clue2/console-ui package (~3 days scaffolding) to prevent React duplication drift
Phase 3 · Second wave + new surfaces Weeks 7–12
  • Wave 2 migrations: ServiceAccounts · ManageDb · Domains · Registries · VPN · Daari SYSTEM surfaces · Heal-agent apply UI
  • Ship P2 new: Incidents & On-Call · Growth Analytics · Cost forecast + anomalies · AWS-tag reconciliation · SLO board
  • Ship P3 new: Staff Tools · declarative control-ns config editor (fixes Platform Config Gap via git-backed source + ArgoCD) · GitHub App migration cutover console · dependency-catalog probes
  • End state: console/src/helpers/constants.js no longer imports USER_TYPE_SYSTEM. Zero hasPermission(["SYSTEM"]) blocks in console. Feature-flag cleanup pass deletes all dashboardOwns.* flags

05 · Open questions

Twelve decisions still on you.

Recommendations in bold. None of these block P1; each shapes P2 or later.

  1. Namespace placement: dashboard in control (shares core-sa + aws-iam-credentials) or new admin namespace (cleaner tenancy boundary)? Recommend control for P1 with a note to revisit at P3.
  2. Impersonation model: scoped short-lived JWT for act-as (useful, higher blast radius) vs read-only view-as (safer, less useful)? Require a second SYSTEM user to co-sign, or ship v1 solo-approved with audit-only? Recommend view-as v1, act-as v2 with co-sign.
  3. Feature flags: in-house table (2 days, capability-limited) vs LaunchDarkly/OpenFeature/Flagsmith (fast to prod, slower to add)? Recommend in-house for P1, revisit if flag count > 50.
  4. Incidents section: own tracker vs integrate PagerDuty/Opsgenie/incident.io? Recommend own for v1 — small team, rabbit hole risk on the integration.
  5. AWS cost granularity: Cost Explorer (daily, 30d lag, easy) vs AWS CUR (hourly, S3+Athena pipeline, correct-long-term)? Recommend Cost Explorer for P1, CUR when forecasting shipped.
  6. Console /costs and /metrics tiles: stay as per-account customer views, or delete because Account settings covers it? Recommend stay — different audience (customer vs staff).
  7. Invitation-flow operator wizard (PRs 2/3/4 pending): build against console or admin? Recommend admin — belongs with staff — but delays it if P1 slips.
  8. Feature-flag flip granularity: per-user (staff dogfoods one week ahead, doubles QA matrix) or global only? Recommend per-user for staff, global for external — cost is one extra query on flag read.
  9. Declarative control-ns config editor (P3): write to coordinator via API, or git repo + ArgoCD reconcile? Git + ArgoCD is the right fix for the Platform Config Gap; adds ~3 weeks to P3.
  10. Runbook execution UI (heal wedged kpack, rotate expired PAT): one-click buttons in the dashboard vs CLI-only? Recommend buttons with dry-run + audit — discoverability wins.
  11. aws-dashboard tab renames when absorbed at P2 (Services/Access/Cluster/Compute): rethemed or preserved verbatim for muscle memory? Recommend preserve. Bookmark churn isn't worth the tidy.
  12. Console soft-redirect banners: sunset at 6 months (less code) or keep forever (nicer UX)? Recommend 6-month sunset per page, tracked in cutover ticket.
